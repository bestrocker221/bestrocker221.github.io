    Unleash the potential of Cloudflare Tunnels

    Setup the Cloudflare Tunnel and publish and protect your app in a second.

    What are Cloudflare Tunnels

    Cloudflare Tunnels offer various useful functionalities such as the possibility to connect your self-hosted, non-exposed apps to the Internet via the Cloudflare network. Not only, you can also connect your private network to its network, but this is not covered in this post.

    The main page for CF Tunnel documentation is here: https://developers.cloudflare.com/cloudflare-one/connections/connect-apps/

    In summary, in order to have a tunnel up and running:

    1. Run the cloudflared binary on your server with a pre-generated token from the web portal.
    2. The daemon will be installed and enabled automatically. Tunnel is up.
    3. Configure everything else from the web portal. (you can do from cli as well)

    From the Zero Trust portal you can add a subdomain for your app and create a new forwarding towards your app (different protocol are supported). Cloudflare will create the DNS record (which is a CNAME to a cfargotunnel Cloudflare zone) and automatically enforce HTTPS with a valid Let’s Encrypt certificate!

    Once setup, Cloudflare will redirect the HTTPS requests to your set domain, towards your self hosted application/address defined above.

    Did I mention that this is included in the free version? This is only the beginning.

    Cloudflare Tunnels offer many benefits:

    • Avoid opening any port on the firewall
    • Automatic DNS and TLS certificate renewal and management
    • Conditional access policies at your control from the dashboard

    Security Considerations

    Cloudflare is a proxy, so they are technically able to see the traffic. Honestly I trust them more than others, but it is to keep in mind that nothing is truly end-to-end encrypted here. (The traffic is always encrypted, from the client up towards your self-hosted application, but it is routed within Cloudflare network proxies)

    Their proxy terminates the incoming TLS connection, can inspect the HTTP plaintext, and then creates a new TLS connection to the destination.

    I am also an open-source passionate, so Cloudflare would introduce something that is closed source (back-end, the cloudflared agent is on GitHub) and not peer-reviewed, although their reputation is top tier.

    This still means that you are running a binary as root in your private environment that has complete network access to your internal network. In an event of Cloudflare being breached, somebody maliciously modifying the agent build pipelines or them “going rogue” (both very unlikely), your private environment is compromised as well.

    A way to restrict this would be to manually restrict the binary permissions on your system, restrict its capabilities, restrict what it can access, firewall it, run it in a “jail”/container/secure environment.

    All of this to say, if you want your data to be 100% private then don’t use Tunnels for that app (ex: a nextcloud/owncloud/photoprism instance).

    How does a Tunnel work

    When the cloudflared daemon connects to Cloudflare network, it will open four QUIC (UDP) connections towards two geographically different Cloudflare data-centers ( four different IPs ) to maintain redundancy and availability.

    The daemon will then listen for instructions, ie: it will open a connection towards your self-hosted address and port only when the app is accessed.

    You can manually check what the process is connecting to with the following command:

    $ sudo netstat -anp | grep -i cloudflared
